## 一、产品定位

### 1.1 核心价值

**CryptoQClaw** — 让每个人都能拥有自己的 AI 量化交易团队

| 维度 | 描述 |
|------|------|
| **产品形态** | 对话式 AI 代理（Telegram/WhatsApp）+ 轻量客户端（敏感信息管理） |
| **目标用户** | 加密货币投资者、量化交易爱好者 |
| **核心价值** | AI 驱动的策略回测 + 自动交易，通过对话完成所有操作 |
| **商业模式** | 免费使用，盈利分润 10%（按结果付费） |

### 1.2 核心理念

> **"对话即操作，随时随地掌控交易"**

**交互模式：**

| 交互渠道 | 用途 | 说明 |
|----------|------|------|
| **Telegram/WhatsApp** | 核心功能交互 | 策略编写、回测、交易、账单、支付等全部通过对话完成 |
| **桌面客户端** | 敏感信息管理 | 仅用于 API Key 配置、交易所密钥等敏感信息 |

**通过对话实现的功能：**
- ✅ 策略编写：用自然语言描述策略，AI 生成代码
- ✅ 数据下载：对话指令下载历史数据
- ✅ 策略回测：对话启动回测，AI 解释结果
- ✅ 模拟交易：对话开启/关闭模拟交易
- ✅ 实盘交易：对话确认后启动实盘
- ✅ 账单对账：对话查看交易明细和账单
- ✅ 计算收费：对话查看本月应付费用
- ✅ 确认收费：对话确认账单金额
- ✅ 二维码支付：对话中生成支付二维码
- ✅ 风控通知：实时推送交易和风控通知

**客户端仅管理：**
- 🔐 大模型 API Key（OpenAI/Claude/本地模型）
- 🔐 交易所 API Key（Binance/OKX）
- 🔐 本地数据加密存储

### 2.2 分润计费模型

#### 高水位机制 (High Watermark)

```
高水位原理：
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  利润曲线                                                   │
│     ╭─────╮        ╭──────────────╮ ← 当前高水位          │
│    ╱       ╲      ╱                ╲                        │
│   ╱         ╲____╱                  ╲                       │
│  ╱                                    ╲ ← 当前利润          │
│ ╱                                      ╲                    │
│─────────────────────────────────────────────────────────────│
│                                                             │
│  计费基础 = max(0, 当前累计利润 - 历史最高累计利润)         │
│  应付费用 = 计费基础 × 10%                                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 示例场景

| 月份 | 当月盈亏 | 累计利润 | 历史高水位 | 计费基础 | 应付费用 |
|------|----------|----------|------------|----------|----------|
| 1月 | +$500 | $500 | $0 | $500 | $50 |
| 2月 | -$200 | $300 | $500 | $0 | $0 |
| 3月 | +$100 | $400 | $500 | $0 | $0 |
| 4月 | +$300 | $700 | $500 | $200 | $20 |
| 5月 | -$100 | $600 | $700 | $0 | $0 |
| 6月 | +$400 | $1000 | $700 | $300 | $30 |

**核心原则：**
- 只有创造**新**的利润才收费
- 亏损后回本不收费
- 永远不让用户为"填补亏损"付费

### 2.3 收入预测（修正版）

#### 保守场景

| 指标 | 数值 |
|------|------|
| 首年目标用户 | 500 |
| 月均留存率 | 85% |
| 人均月利润 | $150 |
| 付费转化率 | 40%（基于诚信，部分用户可能不付） |
| 首月收入 | $150 × 500 × 40% × 50% × 10% = $1,500 |
| 首年收入 | ~$18,000 |

#### 乐观场景（12个月后）

| 指标 | 数值 |
|------|------|
| 活跃用户 | 2,000 |
| 人均月利润 | $200 |
| 付费转化率 | 50% |
| 可计费用户比例 | 60% |
| 月收入 | $200 × 2000 × 50% × 60% × 10% = **$12,000** |
| 年收入 | **$144,000** |

#### 关键假设与风险

| 假设 | 风险 | 缓解措施 |
|------|------|----------|
| 策略能产生稳定收益 | 市场环境变化导致策略失效 | 持续策略优化、多策略组合 |
| 用户愿意支付10%分润 | 用户可能觉得比例过高 | 提供阶梯费率、早期用户优惠 |
| 用户诚信付费 | 本地计费模式下用户可能逃避付费 | 服务价值绑定、社区激励、诚信体系 |
| 加密货币市场持续增长 | 熊市导致交易量下降 | 拓展其他市场、提供熊市策略 |

#### 付费模式说明

本产品采用**本地计费 + 主动支付**模式：

- ✅ 所有计费逻辑在客户端本地完成
- ✅ 用户可查看完整的交易明细和高水位计算过程
- ✅ 支付基于用户诚信，无强制验证
- ✅ 服务端仅接收支付金额和交易哈希，用于确认支付

**为什么采用这种模式？**

1. **隐私保护**：交易数据永不上传，符合"物理级数据隔离"理念
2. **与 WindClaw 一致**：对标竞品的本地优先策略
3. **降低开发成本**：无需复杂的交易验证系统
4. **建立信任**：让用户感受到被信任，反而更愿意付费

由于采用**本地优先**架构，云服务仅承担最小化功能：

| 组件 | 技术选型 | 说明 |
|------|----------|------|

| **启用supabase key** | 对于后来又支付了费用的用户，重新启用其supabase key，并更新用户数据 |

## 十一、产品交付形态

### 11.1 设计原则：本地优先

本产品采用**本地优先（Local-First）**架构，核心理念：

> **"数据在本地，隐私有保障，离线也能用"**

| 原则 | 说明 |
|------|------|

## 十二、安全设计

### 10.1 安全架构

```
┌─────────────────────────────────────────────────────────────┐
│                       安全层级                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Layer 1: 传输安全                                          │
│  ├─ 所有通信使用 HTTPS (TLS 1.3)                           │
│  ├─ 证书固定 (Certificate Pinning)                         │
│  └─ API 请求签名验证                                        │
│                                                             │
│  Layer 2: 认证授权                                          │
│  ├─ JWT Token 认证                                          │
│  ├─ Refresh Token 机制                                      │
│  ├─ 设备绑定 (可选)                                         │
│  └─ 多因素认证 (可选)                                       │
│                                                             │
│  Layer 3: 数据加密                                          │
│  ├─ 敏感数据 AES-256-GCM 加密                               │
│  ├─ 密钥派生 PBKDF2 (100,000 iterations)                   │
│  ├─ 本地存储加密 (SQLite 加密)                              │
│  └─ 内存安全 (敏感数据用后即焚)                             │
│                                                             │
│  Layer 4: 业务安全                                          │
│  ├─ 交易上报 HMAC 签名                                      │
│  ├─ 防重放攻击 (Nonce + Timestamp)                         │
│  ├─ 速率限制                                                │
│  └─ 审计日志                                                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 10.2 数据安全

| 数据类型 | 存储位置 | 加密方式 | 访问控制 |
|----------|----------|----------|----------|
| 用户密码 | 云端 | bcrypt (cost=12) | 仅验证，不可逆 |
| 主 API Key | 云端 + 客户端 | AES-256-GCM | 用户主密钥，用户密码派生密钥 |
| 大模型和交易所 API KEY | 仅客户端 | 本地 AES-256 | 用户密码派生密钥 |
| Supabase Key | 云端下发 + 客户端存储 | AES-256-GCM | 用于下载交易信号、上传收费规则同意、上传账单 |
| 交易记录 | 仅客户端 | 本地 AES-256 | 用户私有数据 |
| 高水位记录 | 仅客户端 | 本地 AES-256 | 用户私有数据 |

### 10.3 密钥管理方案（新增）

#### 密钥层次结构

```
┌─────────────────────────────────────────────────────────────┐
│                     密钥层次结构                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Level 0: 主密钥 (Master Key)                               │
│  ├─ 由用户密码派生 (PBKDF2)                                 │
│  ├─ 不存储，每次从用户密码重新派生                          │
│  └─ 用于加密 Level 1 密钥                                   │
│                                                             │
│  Level 1: 数据加密密钥 (DEK)                                │
│  ├─ 每个用户一个唯一 DEK                                    │
│  ├─ 由主密钥加密后存储                                      │
│  └─ 用于加密实际数据                                        │
│                                                             │
│  Level 1.5: Supabase Key（服务端下发）                      │
│  ├─ 用户注册/登录后由服务端下发                             │
│  ├─ 用于访问 Supabase 服务                                  │
│  │   • 下载每日交易信号（BTC/ETH 策略）                     │
│  │   • 上传收费规则同意记录                                 │
│  │   • 上传用户确认的账单数据                               │
│  ├─ 存储在客户端本地（加密）                                │
│  ├─ 逾期未支付时服务端禁用该 Key                            │
│  └─ 支付后服务端重新启用该 Key                              │
│                                                             │
│  Level 2: 通信密钥                                          │
│  ├─ HMAC 密钥: 用于请求签名                                 │
│  ├─ Session 密钥: JWT 签名                                  │
│  └─ 定期轮换                                                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 架构变更说明 (v6.0)

本版本进行了**交互模式重构**，核心功能从桌面客户端改为通过对话渠道（Telegram/WhatsApp）完成：

| 维度 | v5.0（旧） | v6.0（新） |
|------|-----------|-----------|
| **交互方式** | 桌面客户端为主 | **对话（Telegram/WhatsApp）为主** |
| **策略编写** | 客户端界面 | **对话中自然语言描述** |
| **数据下载** | 客户端操作 | **对话指令** |
| **策略回测** | 客户端界面 | **对话启动，AI解释结果** |
| **模拟/实盘** | 客户端操作 | **对话确认开启** |
| **账单对账** | 客户端界面 | **对话查看** |
| **计算收费** | 客户端查看 | **对话查询** |
| **确认收费** | 客户端点击 | **对话中确认** |
| **支付** | 客户端扫码 | **对话中生成二维码** |
| **客户端用途** | 所有功能 | **仅管理敏感信息（API Key）** |

**核心理念**：
> "对话即操作，随时随地掌控交易"

### v6.0 新增/变更功能摘要

1. **对话式交互（核心变更）**
   - 用户通过 Telegram/WhatsApp 与 AI 代理对话
   - 所有核心功能通过对话指令完成
   - AI 主动推送交易通知和风控告警

2. **客户端精简**
   - 桌面客户端仅用于管理敏感信息
   - API Key 本地加密存储，永不上传
   - 简化客户端开发复杂度

3. **收费规则同意（对话中）**
   - 机器人推送收费规则卡片
   - 用户点击按钮同意
   - 同意结果上传服务器存档

4. **月度计费（对话中）**
   - 机器人每月主动推送账单
   - 用户对话中查看明细、确认账单
   - 确认后上传服务器生成支付订单

5. **支付（对话中）**
   - 对话中选择支付方式
   - 机器人发送支付二维码图片
   - 服务器监控链上自动核对
   - 核对成功后机器人推送确认通知
